## Answer Key

**A1.** Records have `private final` fields generated by the compiler — there are no setter methods, and the fields are declared `final` so they literally cannot be reassigned after construction. The compiler generates: `private final int x; private final int y; public Point(int x, int y)` (canonical constructor), `public int x()`, `public int y()`, `equals()`, `hashCode()`, `toString()`. The `final` modifier is the enforcement mechanism.

**A2.** The `switch` is missing the `case Pending p ->` branch. With sealed types, the compiler enforces exhaustiveness — every permitted subtype must be covered. Add `case Pending p -> ...`. If you genuinely want to ignore `Pending`, add `default -> throw new IllegalStateException("Unexpected: " + result)` — but then you lose compile-time exhaustiveness checking.

**A3.** Python's `asyncio` uses cooperative multitasking — you must explicitly `await` every I/O operation, which suspends the coroutine and returns control to the event loop. Java virtual threads use preemptive unmounting — when a virtual thread blocks on I/O (even using normal blocking API like `socket.read()`), the JVM automatically unmounts it from the OS (carrier) thread without any code change. The key difference: virtual threads work with existing blocking code; asyncio requires rewriting all I/O as coroutines.

**A4.**
```java
sealed interface PaymentResult permits Ok, NotFound, RateLimited, ServerError {}
record Ok<T>(T data) implements PaymentResult {}
record NotFound(long id) implements PaymentResult {}
record RateLimited(Duration retryAfter) implements PaymentResult {}
record ServerError(String message) implements PaymentResult {}

// Handling:
return switch (result) {
    case Ok(var data) -> ResponseEntity.ok(data);
    case NotFound(var id) -> ResponseEntity.notFound().build();
    case RateLimited(var retry) -> ResponseEntity.status(429).header("Retry-After", retry.toString()).build();
    case ServerError(var msg) -> ResponseEntity.internalServerError().body(msg);
};
```
Benefits over Optional: carries rich data per case. Better than exceptions: no stack trace overhead for expected failures. Compiler enforces handling all cases.

**A5.** Enable Spring Boot 3.2 virtual threads: `spring.threads.virtual.enabled=true` in `application.properties`. This replaces Tomcat's thread pool with virtual-thread-per-request. No code changes needed. Instead of 200 platform threads (capped at 200 concurrent requests), the JVM creates one virtual thread per request. With 10ms DB latency and 10,000 concurrent requests: old model blocks 9,800 requests; virtual thread model handles all 10,000 concurrently on ~CPU-count carrier threads.

**A6.** New code:
```java
return switch (obj) {
    case String s -> s.toUpperCase();
    case Integer i -> i * 2;
    case null -> throw new NullPointerException("obj is null");
    default -> throw new IllegalArgumentException("Unexpected: " + obj.getClass());
};
```
Old code with `null`: `obj instanceof String` returns `false` for null — falls through all branches, returns null (or NPE if you dereference). New switch with `case null` explicitly handles it — no silent fall-through.

**A7.** No null check on `name`. If `name` is null, `name.trim()` throws NPE inside the compact constructor. Add: `Objects.requireNonNull(name, "name must not be null")`. Also, `age = age` is a no-op in a compact constructor — you don't reassign parameters, the compiler does it. The validation should be: `if (age < 0) throw new IllegalArgumentException("age must be >= 0")`.

**A8.** The `default -> 0` case defeats exhaustiveness checking. With `sealed interface Shape permits Circle, Rectangle`, the switch is already exhaustive without `default`. The problem: if you add `Triangle` to the sealed class, the compiler will NOT warn you that this switch doesn't handle `Triangle` — the `default` silently returns 0 for Triangle shapes. Remove `default` to get compile-time exhaustiveness checking.

**A9.** Bug 1: `sharedCounter++` is not atomic — virtual threads run on OS threads in parallel, so this is a race condition. Fix: use `AtomicInteger`. Bug 2: The lambda captures `sharedCounter` which must be effectively final for a lambda — this won't compile. Fix: use `AtomicInteger counter = new AtomicInteger(); executor.submit(() -> processRequest(counter.getAndIncrement()))`.

**A10.**
```java
sealed interface OrderState permits Draft, Submitted, Confirmed, Shipped, Delivered, Cancelled {}
record Draft(OrderId id, List<LineItem> items) implements OrderState {}
record Submitted(OrderId id, Instant submittedAt) implements OrderState {}
record Confirmed(OrderId id, Instant confirmedAt, String warehouseId) implements OrderState {}
record Shipped(OrderId id, String trackingNumber, Instant shippedAt) implements OrderState {}
record Delivered(OrderId id, Instant deliveredAt) implements OrderState {}
record Cancelled(OrderId id, String reason, Instant cancelledAt) implements OrderState {}

// State transition:
OrderState confirm(OrderState state, String warehouseId) {
    return switch (state) {
        case Submitted s -> new Confirmed(s.id(), Instant.now(), warehouseId);
        case Draft d -> throw new IllegalStateException("Must submit first");
        default -> throw new IllegalStateException("Cannot confirm from state: " + state.getClass().getSimpleName());
    };
}

// Business logic routing:
double calculateShippingCost(OrderState state) {
    return switch (state) {
        case Draft d -> estimateFromItems(d.items());
        case Confirmed c -> fetchActualRate(c.warehouseId());
        case Shipped s, Delivered s2 -> 0.0;  // already shipped
        case Cancelled c -> 0.0;
        case Submitted s -> throw new IllegalStateException("Not yet confirmed");
    };
}
```
